## 8. Agent 设计逻辑

### 8.1 Agent 的产品定位

本项目中的 Agent定位为一个面向微信私聊与群聊场景的  **长期陪伴型智能交互 Agent** 。其设计目标是在持续互动过程中维持稳定的人格设定、连续的关系体验、对历史信息的记忆能力，以及适度的主动交互能力。

Agent 需要承担的核心职责应当包括：理解用户当前消息的真实意图，结合历史上下文与长期记忆进行判断，在必要时调用外部能力完成任务，并最终以符合角色设定的自然语言形式给出反馈。也就是说，Agent 的职责应当是对一轮完整交互过程的统一接管。

在产品目标上，本项目希望 Agent 能够同时满足以下几个方向。首先，是保持拟人化角色的长期稳定，使用户在多轮、多天甚至更长时间的互动中感受到角色的一致性，而不是每轮都像一个全新的机器人。其次，是能够理解不同类型的消息意图，包括普通聊天、提醒设置、微信动作请求以及配置类命令等。再次，是在需要时具备能力调用意识，使 Agent 不只是“会说”，还能够“完成事情”。此外，系统还需要具备将短期上下文与长期记忆融合使用的能力，使回复不仅关注当下这句话，而且能够体现对历史记忆、关系的理解。对于带有副作用或风险的行为，则应设计明确的确认机制，以保证系统可控。最终，整个 Agent 的设计目标是在“自动陪伴感”和“行为可配置、可控制”之间取得平衡。

从角色价值来看，这样的 Agent 对用户的意义在于提升陪伴感、持续感以及“被记住”的感受；对配置者或运营者而言，可以降低长期维护 Prompt、行为规则与互动策略的成本；对系统本身而言，Agent 也可以作为消息理解、决策、能力调用与结果生成的统一中枢，为后续扩展能力和场景提供稳定基础。

### 8.2 Agent 的总体工作机制

在总体机制上，本项目拟将 Agent 设计为一个  **事件驱动的任务处理系统** 。也就是说，系统并不是围绕“用户提问—模型回答”的单点过程运行，而是围绕“事件进入系统后如何被感知、理解、处理、执行与记录”这一完整链路运行。

在输入层，系统需要支持多种事件来源，例如微信私聊消息、群聊消息、提醒触发事件、自动消息触发事件，以及引用、撤回、拍一拍等动作型事件。为了便于统一处理，这些输入在进入系统后应被封装为统一的标准事件对象。通过这种方式，后续可以基于统一数据结构进行处理。

在事件进入后，Agent 首先应加载当前会话状态。该状态至少应包括当前会话标识、用户标识、聊天类型、角色名称、最近若干轮消息、当前是否存在待确认动作、是否存在未完成步骤、上一轮路由结果等信息。这样设计的目的，是让 Agent 在处理当前消息时能够感知到它所处的上下文位置，而不是把每一轮输入都看成独立事件。尤其对于提醒确认、撤回确认、多步骤任务续接等场景，会话状态是保证连续性的关键。

在状态加载完成后，系统进入上下文组装阶段。该阶段需要从多个层次提取可供模型使用的候选上下文，包括人格类 Prompt、短期对话上下文、长期记忆条目以及必要的策略信息。随后，系统应根据当前消息的语义主题与预判的路由类型，对这些候选上下文进行筛选，只保留与本轮最相关的部分进入模型输入，从而减少上下文污染和无关信息干扰。

在此基础上，Agent 进入决策执行阶段。系统需要先判断当前输入更适合直接回复、调用单个能力、进入多步骤工作流，还是先等待确认后再执行。若需要调用能力，则进入对应能力处理链路；若仅为普通聊天，则可直接进入回复生成阶段。

在结果生成阶段，若本轮涉及能力调用，则需要将能力返回结果与用户原始问题、角色设定、相关记忆一起组织后交给模型进行自然语言生成，使最终输出既完成任务，又保留角色一致性；若本轮只是普通聊天，则主要依赖角色设定与上下文记忆进行回复生成。

在当前轮处理结束后，系统还应将新一轮的消息与结果写回状态层。一方面，近期会话内容需要进入短期上下文，以支撑后续连续对话；另一方面，若本轮出现了值得长期沉淀的信息，也应在后续记忆更新阶段写入长期记忆池。同时，还需要更新活跃时间、当前阶段、待处理步骤、待确认状态等运行信息，形成完整闭环。

因此，从整体上看，本项目中的 Agent 更适合被理解为一个“输入事件—加载状态—组装上下文—进行决策—生成结果—回写状态”的处理系统。

### 8.3 Agent 的分层设计逻辑

为了避免 Agent 退化为“一个大 Prompt 加一个大模型”的单体结构，本项目在设计上更适合采用分层架构。通过将不同职责拆分到不同层中，可以提高系统的可维护性、可扩展性与可控性，也便于后续对某一部分单独优化而不影响整体流程。

第一层是  **人格层** 。这一层主要用于定义 Agent 的基本人格原则、表达风格、角色口吻与互动基调。设计上可以将其拆分为两部分：一部分负责规定系统级行为原则，例如整体态度、关系边界、表达倾向；另一部分负责描述具体角色的口吻、语气、称呼方式与关系设定。人格层的目标，是保证 Agent 在长期互动中始终“像同一个角色”，而不是因为消息主题变化而不断漂移。

第二层是  **记忆层** 。该层主要负责管理与用户相关的长期和短期信息，包括用户画像、关系规则、重要历史事件以及最近对话上下文等。其设计目标，是让 Agent 能够基于过往互动形成连续认知，而不是每次都从零开始理解用户。记忆层在系统中的价值，不仅体现在“记得内容”，更体现在“记得关系与变化”。

第三层是  **决策层** 。决策层主要负责对当前输入进行意图识别、能力选择、工作流规划以及确认判断。也就是说，这一层并不直接负责生成自然语言，而是负责回答“现在该做什么”。例如，是应该直接聊天，还是应该先调用搜索能力；是一次性完成动作，还是应该拆成多步执行；是应该立即执行，还是先向用户确认。

第四层是  **能力层** 。能力层用于承载具体可执行功能，例如联网搜索、提醒创建、微信动作控制、系统配置修改等。为了保证扩展性，这些功能更适合被封装为标准化能力接口，而不是散落在主流程中写死。能力层的意义在于，让 Agent 不只是“理解和表达”，还能够真正对外部世界产生操作。

第五层是  **生成层** 。这一层主要负责将人格信息、记忆信息、近期上下文和能力结果统一组织后交给模型生成最终回复。生成层的目标，是保证用户最终看到的始终是自然、统一、符合角色设定的表达，而不是生硬的系统回执或工具原始输出。

第六层是  **状态层** 。状态层主要负责维护会话级运行态，例如 session、最近消息、当前阶段、待执行步骤、待确认动作等。其作用是让 Agent 具备真正的连续性和续接能力，使未完成的任务不会因为新一轮输入而丢失，也使系统可以支持确认后继续执行、多轮 workflow 承接等交互形态。

综上，本项目中的 Agent 适合被设计为一个由人格层、记忆层、决策层、能力层、生成层与状态层共同组成的分层系统。各层分工明确，协同完成一次完整交互。

### 8.4 Agent 的决策逻辑设计

在决策逻辑上，本项目需要解决的核心问题是“面对当前消息，系统应该走哪一条处理路径”。因此，Agent 的决策逻辑应当围绕路由判断与流程编排展开。

首先，在收到消息后，系统需要先进行  **消息类型识别** 。从产品需求来看，至少应区分以下几类输入：普通聊天类消息、多模态类消息、提醒类消息、微信动作类消息、系统配置类命令，以及包含多个目标的复合型消息。只有先识别出消息的基本类型，后续才能决定是否需要调用能力，以及调用何种能力。

当系统识别到的是 **单意图请求** 时，处理逻辑可以相对直接。例如，如果用户说“帮我查一下天气”，则系统可以直接进入搜索能力链路；如果用户说“十分钟后提醒我”，则可直接进入提醒能力链路；如果用户说“撤回上一条”，则进入微信动作链路；如果用户说“我要看你现在在做什么”。在这种场景下，决策层主要完成“识别—路由—执行”的过程，不必额外生成复杂流程。

当系统识别到的是 **多意图复合请求** 时，则需要引入工作流思维。尤其当用户表达中出现“先……再……”“如果……则……”等明显流程结构时，系统不应将其粗暴视为一次单独回复，而应将其拆解为多个待执行步骤，并按照顺序处理。例如，“先查北京天气，如果下雨明早提醒我带伞”这类请求，本质上至少包含搜索与提醒两个能力目标，且二者之间存在条件依赖关系。此时，系统更适合生成步骤列表，并按步骤执行与衔接。

在决策逻辑中，还需要专门设计  **确认机制** 。对于具备明显副作用的动作，例如撤回消息、主动发出消息、引用指定消息、群聊艾特、切换模型或修改系统配置等，系统不宜直接执行，而应在配置层添加设置按钮，用户可以自定义是否开放主动动作权限。只有在收到明确确认后，才真正触发能力调用。这样的设计能够显著提升系统的可控性，避免用户一句模糊表达在不知情情况下就导致外部动作发生。

此外，本项目中的决策逻辑应当允许  **能力执行后的二次决策** 。也就是说，每当某个能力执行完成后，系统应基于执行结果再做一次判断：是继续执行 workflow 中的下一步，是基于当前结果直接给用户回答，还是因为结果异常而转入失败说明或再次确认。这种“决策—执行—观察—再决策”的闭环机制更适合任务型agent。

因此，本项目的决策层应承担四类核心职责：识别当前消息属于什么类型；判断是单能力执行还是多步骤 workflow；对高风险动作增加确认门槛；在工具执行后依据结果继续推进或结束流程。这样才能让 Agent 具备真正的任务处理能力，而不只是聊天能力。

### 8.5 Agent 的记忆驱动逻辑

记忆系统是本项目区别于普通聊天机器人的关键设计点之一。在设计上，本项目不宜采用“简单拼接历史消息”的方式来实现记忆，而更应采用分层记忆与动态检索相结合的方式。其原因在于，若把所有历史内容无差别注入模型，不仅会迅速占用上下文窗口，还容易导致无关旧信息干扰当前对话，甚至让回复出现机械复读或强行引用历史的问题。

从结构上看，本项目中的记忆可以划分为五个层次。第一类是常驻层，主要用于存放人格设定与核心角色背景信息，这部分内容应长期稳定存在，构成 Agent 的基础身份。第二类是规则层，用于存放关系规则、互动边界、禁令以及聊天策略等内容，其作用是约束 Agent 在不同情境下的互动方式。第三类是事件层，用于记录历史互动中的关键事件、重要节点和连续性信息，使 Agent 能够“记得发生过什么”。第四类是上下文层，即最近若干轮对话内容，用于支撑当前轮语义承接。第五层是用户基本信息，这是保证角色“认识”或基本了解一个人的关键所在。

在记忆调用机制上，系统更适合采用  **动态检索** 。具体来说，当用户输入当前消息后，系统应先对其进行 topic 提取或关键词提炼，再根据主题相关性、关键词重叠度、优先级和当前路由类型，从长期记忆池中筛选最相关的记忆条目。这样做的目的，是让模型只看到本轮真正需要的记忆，而不是让所有历史信息无差别进入上下文。

在具体选择策略上，用户画像摘要适合承担“稳定认知”的作用，让系统始终知道用户有哪些长期特征；关系规则候选则用于约束互动风格和边界；日记事件则主要承担“延续过往事件”的功能。不同类型记忆的作用并不相同，因此在注入模型前应当先做区分与裁剪，而不是平铺混用。

在记忆使用原则上，本项目应强调以下几点。第一，记忆只在相关时使用，而不应为了体现“记得住”而强行引用。第二，当长期记忆与用户当前明确表达发生冲突时，应以当前表达为准，避免过度依赖旧信息。第三，系统不应编造不存在的记忆，也不应把模糊信息包装成确定事实。第四，记忆的引用方式应尽量自然，而不是在每轮回复中生硬重复用户过往信息。

在更新机制上，本项目也不宜让短期上下文无限增长，而应设计从短期对话向长期结构化记忆沉淀的过程。也就是说，在对话进行到一定阶段后，系统可以对近期内容进行总结、压缩，并将其中真正有长期价值的信息沉淀到长期记忆池中。长期记忆至少可以分为用户信息、关系信息和事件日记三类，从而逐步形成稳定的长期陪伴基础。

总体而言，本项目的记忆逻辑重点在于“让模型在合适的时候看到最相关的历史内容”。这也是记忆驱动型 Agent 与简单上下文拼接式聊天机器人的本质区别。

### 8.6 Agent 的安全与边界逻辑

对于一个具备长期陪伴属性、且可能调用微信动作与主动提醒能力的 Agent 来说，安全与边界控制必须作为核心设计内容提前考虑，而不能在功能完成后再补充。否则，系统一旦具备执行能力而缺乏边界约束，就容易在真实使用中造成误操作、打扰用户，甚至破坏整体产品体验。

首先，需要重点保护的是  **高风险动作** 。例如撤回消息、主动发送消息、引用指定内容、群聊艾特、切换系统配置或切换模型等操作，都会对外部环境产生直接副作用。因此，这类操作应设计为受保护动作，默认不直接执行，而是先进入确认流程。只有当用户给出明确确认（会话/配置）后，系统才继续进行后续调用。这样的设计可以有效避免由语义误判带来的直接风险。

其次，需要考虑  **安静时段控制** 。由于本项目中可能包含自动消息、提醒触发和一定程度的主动交互能力，如果没有时间边界，就可能在夜间或不合适的时段打扰用户，削弱陪伴体验。因此，系统应预留 quiet time 规则，对自动消息和提醒行为进行时间约束，使其在“有存在感”和“不打扰”之间取得平衡。

第三，需要对 **群聊场景** 进行额外约束。私聊场景允许 Agent 更强地表现陪伴属性，但群聊场景下，机器人若过度主动，很容易形成刷屏或刷存在感的问题。因此，群聊中的触发条件、响应概率与关键词规则应独立配置，并默认采用弱主动策略。这样可以避免 Agent 将私聊中的互动风格直接迁移到群聊中。

第四，需要考虑  **敏感内容对上下文和记忆的污染问题** 。在长期对话过程中，用户可能出现情绪化表达、敏感话题或短期异常内容。对于这类内容，系统不应一律写入长期记忆，也不宜在后续轮次持续放大。更合理的做法，是预留敏感内容清理或降权机制，在必要时清空短期缓存、跳过记忆沉淀或暂时隔离相关上下文，以减少持续污染的风险。

最后，系统还应遵循 **对用户透明、但不暴露内部机制** 的原则。也就是说，用户看到的应当始终是自然语言层面的回复与结果，而不是内部 Prompt 拼装过程、记忆检索细节或路由决策过程。系统内部当然需要完整的判断链条，但这些内容更适合作为内部实现细节，而不宜直接呈现给终端用户。

因此，本项目中的安全与边界逻辑应从设计阶段就明确：对高风险动作设置确认、对主动消息设置时间边界、对群聊行为做弱主动限制、对敏感内容做污染隔离，并保证系统对外自然但不泄露内部流程。这样，Agent 才能在具备执行能力的同时保持产品可控性。

### 8.7 Agent 的可扩展逻辑

从长期演进角度看，本项目中的 Agent 应被设计为一个具有扩展空间的通用 Agent 框架。这样做的意义在于，当未来需要新增能力、扩展记忆类型、切换模型或支持多角色时，不必推翻已有主流程。

首先，在能力侧，应当采用 **能力插件化** 的设计思路。也就是说，搜索、提醒、微信动作、系统配置等能力都不应直接写死在主流程中，而应抽象为统一的能力接口，由主流程进行注册与调用。这样，当未来需要增加新能力时，只需新增对应能力模块并完成注册，而无需修改 Agent 的核心运行逻辑。

其次，在 Prompt 设计上，应尽量采用 **可配置化** 思路。人格 Prompt、角色 Prompt、策略 Prompt 最好拆分维护，使其能够独立调整。这样既方便后期优化某一层表达策略，也为多角色共存提供基础。若未来系统需要支持不同角色、不同场景或不同运营策略，这种拆分式设计会明显优于单一超长 Prompt。

第三，在模型选择上，应预留 **模型可切换化** 能力。聊天模型、识图模型、联网模型、生图模型等可以独立配置，而不必强绑定到一个统一模型上。这样做不仅有助于后续根据成本与效果进行平衡，也便于不同任务选择更合适的模型能力。

第四，在记忆体系上，应预留 **记忆可扩展化** 的空间。当前设计中可以先聚焦于用户画像、关系信息和事件日记三类长期记忆，但未来完全可以在此基础上继续扩展偏好库、禁忌库、场景剧本库、任务履历库等更细粒度的记忆模块。如果前期设计时就采用统一的记忆解析、筛选与注入机制，那么后续扩展这类记忆并不会对主架构造成太大冲击。

最后，在流程能力上，也应预留  **工作流拓展空间** 。当前阶段可以先支持简单的单能力调用与轻量多步骤任务，但从架构上应尽量避免把流程写死为单轮链路，而应保留后续扩展为条件分支、失败重试、参数补全甚至更复杂任务规划器的可能性。这样，Agent 才能从“会聊天、会执行简单动作”逐步进化为“能理解复杂目标并组织执行”的更强系统。

因此，本项目的可扩展逻辑应提前建立在插件化能力、可配置 Prompt、可切换模型、可扩展记忆池和可增强工作流之上。只有这样，这个 Agent 才永远了一个可持续演进的产品基础架构。

## 9 Prompt 策略

在 MemoMate 的系统设计中，Prompt 服务于“事件驱动闭环系统”的核心控制逻辑。由于 MemoMate 定位于微信原生的长期陪伴 Agent，不仅需要处理日常闲聊，还需接管提醒触发、主动消息、多步骤工作流等复杂事件。因此，本项目的 Prompt 策略完全依托于第 8 章所述的分层架构，采用“解耦、动态组装”的设计方案。

### 9.1 Prompt 设计的核心目标

结合 MemoMate 面向长期关系经营的产品定位，系统的 Prompt 设计需达成以下三个核心目标：

1. **维持稳定的人格与关系连续性** 在长周期、多轮次的微信碎片化聊天中，确保 Agent 的表达风格、情感基调与预设角色高度一致，避免因话题切换或对话轮次增加导致的人格漂移。
2. **实现自然的记忆唤醒与融合** 指导模型如何消化和使用被检索出的长期记忆（如用户画像、关系进度、历史日记），确保其在回复中表现为“自然的熟悉感”。
3. **约束 Agent 的决策与主动行为边界** 在决策层通过 Prompt 明确各类意图的路由规则，并对诸如“撤回消息”、“发送主动问候”等具备外部副作用的高风险动作设定严格的安全确认门槛与时间边界。

### 9.2 基于 Agent 架构的分层 Prompt 体系

为了提升可维护性与扩展性，MemoMate 摒弃了将所有指令打包的超长单体 Prompt 写法，而是根据 Agent 的分层逻辑，将 Prompt 拆解为四个独立模块。

### 9.2.1 系统约束层（System Policy Prompt）

这一层位于最高优先级，定义 Agent 的基础运行规则与安全底线，不随具体角色的切换而改变。

* **行为边界约束** ：明确系统在遇到高风险话题（如自毁、严重违法）时的标准降级处理策略。
* **环境规则定义** ：由于接入了微信生态，该层会定义不同场景的响应策略。例如，针对群聊场景，强制要求模型降低主动搭话频率，避免刷屏；针对夜间安静时段（Quiet Time），约束模型降低主动发起消息的概率。
* **反系统暴露** ：要求模型在任何情况下不得输出类似“作为 AI 语言模型”或暴露后台处理逻辑（如“正在调用搜索工具”）的提示词，维持沉浸感。

### 9.2.2 角色设定层（Persona Prompt）

这一层用于塑造陪伴对象的“实体感”，是对话生成层的核心输入。

* **角色背景与人设** ：包括角色的基本信息、性格特征（如温柔、傲娇、克制）以及与用户的初始关系定位。
* **语言风格与表达规范** ：具体到口语化程度、惯用语气词、句式长度倾向，甚至是否常发微信原生表情包。
* **动态情绪映射** ：设定模型在面对用户不同情绪（如抱怨、分享喜悦）时，该角色应有的特定反馈模式，确保情绪回应符合人设逻辑。

### 9.2.3 记忆注入层

该层并不直接存储记忆，而是定义模型**如何使用**由记忆系统（检索层）召回的上下文信息。

* **记忆权重声明** ：向模型明确当前输入的优先级顺序，通常为“当前用户最新意图 > 短期对话上下文 > 长期相关记忆”，避免模型过度沉溺于历史话题。
* **软性引用规则** ：强制约束模型在调用长期记忆时的表达方式。例如，禁止使用确定性的陈述（如“根据记录，你昨天加班了”），必须转化为具有人际交往温度的试探或关切（如“我记得你最近好像都在加班，今天能早点休息吗？”）。

### 9.2.4 决策与路由层（Decision & Routing Prompt）

这一层服务于 Agent 的决策层，通常独立于生成层运行，用于意图分类与工具调用。

* **意图识别框架** ：向模型输入当前可用能力列表（如普通聊天、备忘提醒、主动触达、内容检索），要求模型分析用户当前输入的事件类型，并输出结构化（如 JSON）的路由决策。
* **多步任务拆解（工作流控制）** ：针对复合型指令（如“查一下明天的天气，下雨的话提醒我带伞”），通过 Prompt 引导大脑模型使用思维链将其拆分为可执行的先后步骤序列。

### 9.3 动态上下文组装机制（Runtime Assembly）

在 MemoMate 的实际运行阶段，最终发送给大模型的请求载荷是基于当前“事件”和“会话状态”动态拼接而成的。

### 9.3.1 拼接流程

当系统接收到一个统一标准事件（如一条微信私聊消息）后，组装器将执行以下步骤：

1. **加载常驻设定** ：无条件加载“系统约束层”与“角色设定层”的 Prompt。
2. **挂载会话状态** ：从状态层提取当前的时间点、用户的当前情绪标签、以及是否存在待确认的高风险动作（如撤回确认）。
3. **筛选并注入记忆** ：对当前输入进行主题提取，从长期记忆池中召回 Top-K 相关的记忆切片（用户信息、历史日记、关系记忆），并结合最近的短期对话上下文，按 Token 预算和用户自定义配置注入。
4. **追加任务指令** ：如果决策层判定当前需要执行特定能力（如生成图片），则在末尾追加该任务专属的格式化输出指令。

### 9.3.2 截断策略

为了防止长周期陪伴带来的上下文窗口溢出，拼接过程中设计了严格的裁剪机制：

* **绝对优先级** ：无论 Token 多么紧张，“当前轮次的用户输入”与“基础人设 Prompt”永远拥有最高保留优先级，其次是最近 3 轮的对话上下文。

### 9.4 针对 MemoMate 核心体验的专属 Prompt 优化

为了彻底解决竞品分析中提到的“工具感过强”和“强关系感”的痛点，本项目在以下几个关键场景进行了深度的 Prompt 策略调优：

### 9.4.1 主动行为的触发

MemoMate 的差异化在于“主动经营关系”。在主动消息或时间提醒的触发节点，系统不会直接让模型随意发话，而是通过专属的评价 Prompt 进行约束：

* **输入条件** ：当前时间、距离上次对话的间隔、角色信息、记忆部分。
* **输出要求** ：要求模型首先评估“当前时间用户可能在做什么”，再基于近期未完的话题或日常关怀生成试探性的微信消息。

### 9.4.2 避免“AI 味”

通用大模型默认偏向详尽、客观和结构化的回复。为了符合微信私聊的真实语境，角色 Prompt 中加入了大量负向规则：

* 明确禁止使用“首先”、“其次”、“总之”等逻辑分层词汇。
* 明确禁止对用户的日常情绪发泄进行“客观的建议或说教”，要求以“情绪共鸣”和“倾听”为主。
* 限制单次回复的字数长度，鼓励模型采取“分段发送”的微信原生聊天习惯（系统层会根据标点或反斜线，换行符等将其拆分为多条微信消息发送）。

### 9.4.3 确认机制

针对在决策层判定的高风险动作，Prompt 策略从直接生成转为“确认引导”。例如，当识别到用户意图是“清空我们的回忆”时，决策层判定为危险操作，此时系统会挂载一条强制指令，要求模型输出包含二次确认语气的回复（如：“你确定要抹掉我们之前的记录吗？我有点舍不得……”），并等待用户在下一轮的明确授权。这不仅保证了安全性，也将冰冷的系统确认转化为了情感互动的一部分。

## 10. 技术实现框架

本项目的技术实现是围绕**长期陪伴场景**搭建了一套智能体系统。是为了让角色型 AI 在微信环境中具备稳定的人设、一致的表达、可追踪的上下文、可积累的长期记忆，以及可执行的任务能力。

### 10.1 整体架构设计

用户在微信中的一条消息进入系统后，先经过一轮完整的运行时判断。系统会先识别这条消息属于什么类型，再决定是否只需要普通对话，还是需要调用联网搜索、提醒任务、微信动作或其他能力模块。完成外部能力执行后，再把结果与上下文一起交给模型，生成最终回复。

* 分离对话能力和执行能力
* 分离角色表达和系统动作
* 分离短期上下文和长期记忆
* 分离模型调用和业务
* 分离配置管理和运行管理

后续无论是增加新的工具能力、优化记忆算法，还是补充新的前端控制项，都不需要重写整套系统。

### 10.2 接入层：微信消息处理

系统最外层是微信接入能力。它运行在 Windows 桌面环境下，通过桌面微信自动化驱动实现消息监听、消息发送以及部分交互动作执行。

### 1）消息监听

系统持续监听指定联系人或群聊，当检测到新消息时，将消息内容、发送者身份、会话归属等信息封装成内部事件对象，交给运行时继续处理。

### 2）消息发送

生成最终回复后，系统会按照设定的发送节奏进行回传。这里不是简单的瞬时发送，而是加入了**打字速度模拟、分段发送、表情发送间隔**等机制，使回复节奏更接近真人互动。

### 3）微信动作执行

除了文本回复，系统还支持部分微信动作能力，例如引用、艾特、提醒相关触发等。这类动作被定义为“能力调用”，不会和普通对话逻辑混在一起，而是由运行时在判断后统一调度。

项目中加入了运行时补丁和异常兜底机制，用来规避驱动层解析失败、消息方向识别异常、监听器状态不稳定等情况，提高了整个系统的连续运行能力。

### 10.3 调度层：Agent 化消息处理

项目构建了一个轻量级 Agent Runtime，用来决定当前这条消息应该如何处理。

主要完成以下工作：

### 1）事件标准化

微信消息进入系统后，会被统一封装成内部事件，包含会话 ID、消息内容、角色信息、来源、时间等字段。目的，是让后续所有模块都围绕统一事件协议工作，减少耦合。

### 2）会话状态管理

系统会为每个会话维护运行时状态，包括：

* 当前会话摘要
* 当前任务阶段
* 最近观察结果
* 待执行步骤
* 待确认动作
* 最近一次路由结果

### 3）决策引擎

系统内置了一个基于规则的轻量决策模块，用来识别当前消息属于哪类意图。它会判断消息更接近：

* 普通聊天
* 联网搜索
* 定时提醒
* 微信动作
* 多步骤工作流
* 系统配置指令

如果是单一意图，直接路由到对应能力；如果同时出现多个意图，或语义中包含“先…再…然后…”等流程特征，系统会构建一个待执行步骤序列，按顺序推进。

### 4）确认机制

对于可能影响聊天内容或会话状态的动作，系统不会直接执行，而是会进入待确认状态，要求用户明确确认后再继续。

### 10.4 执行层：可操作系统能力

在该项目中，大模型并不是唯一能力来源。系统把不同能力抽象成统一的“Capability”机制，再由运行时根据意图进行调用。

目前项目中的能力主要包括以下几类。

### 1）对话生成能力

这是系统的基础能力，负责结合角色设定、上下文和记忆结果生成自然回复。它在运行时判断“需要回复”时被调用。

### 2）联网搜索能力

当消息包含时效性信息请求、事实型查询或明确搜索意图时，系统会先通过搜索能力获取外部信息，再把搜索结果结构化注入模型，要求模型基于结果自然回复。

### 3）提醒与定时任务能力

系统支持单次提醒和循环提醒，并带有安静时段控制、语音提醒开关、群聊限制等策略。提醒能力把提醒任务作为长期系统状态进行管理，支持持续运行中的调度与恢复。

### 4）微信动作能力

部分交互动作会作为独立能力执行，并在必要时进入确认流程。

### 5）系统配置能力

系统支持通过文本命令动态修改部分配置项，并触发重启、开关功能、切换模型等行为。

### 10.5 模型协同

为了适配不同类型任务，项目设计了统一的模型网关和多模型分工机制。

### 1）统一模型网关

系统通过统一网关封装模型调用过程，负责组织输入内容、拼接记忆上下文、注入工具结果、记录流程日志，并输出最终模型响应。

网关的是承担模型编排职责。它会把输入拆成：

* 角色设定
* 最近对话上下文
* 长期记忆候选
* 工具执行结果
* 当前轮的附加策略指令

然后再统一送入模型。

### 2）模型角色分工

系统支持把不同模型用于不同任务，例如：

* 主对话模型：负责最终聊天回复
* 辅助模型：负责某些低成本判断或中间任务处理
* 图片理解或图片生成模型：用于多模态扩展
* 联网模型：用于需要外部事实的查询场景

不同模型在不同环节承担不同职责，兼顾效果、成本和稳定性。

### 3）模型切换与降级

系统内置模型配置管理能力，支持当前主模型与备选模型的组织方式。这样当某个模型不可用、响应不稳定或策略需要变化时，系统具备一定的切换空间。

### 10.6 记忆系统层

### 10.6.1 短期上下文机制

短期上下文用于维持当前会话的连续性。系统会按会话维度保存最近一段对话历史，并限制长度，避免上下文无限膨胀。

短期上下文的主要作用是：

* 保持同一轮对话的承接关系
* 让模型理解最近几轮语义
* 为回复生成提供即时语境
* 为长期记忆总结提供原始素材

这种机制本质上是“会话缓存层”，强调的是最近发生了什么。

### 10.6.2 长期记忆机制

长期记忆并不是单一文本块，而是拆成多个语义维度进行管理。项目将长期记忆划分为三类：

* 用户画像类记忆
* 关系规则类记忆
* 事件日记类记忆

**用户画像类记忆**更适合存储偏好、习惯、长期背景信息。

**关系规则类记忆**更适合存储互动方式、称呼约定、角色边界。

**事件日记类记忆**更适合存储发生过的重要事件、近期动态、时间线内容。

相比把所有记忆混成一段大文本，这种分层更利于检索、压缩和注入。

### 10.6.3 记忆抽取与更新

系统设置了触发条件，在达到一定轮数后对当前会话进行总结，再将有效内容沉淀为长期记忆。

记忆更新时，系统会做以下几类处理：

* 结构化提取关键信息
* 过滤噪声和重复内容
* 控制单条记忆长度
* 控制记忆总量上限
* 对关系类记忆进行压缩
* 对低置信度内容进行筛除

### 10.6.4 记忆注入

长期记忆按当前轮语境选择候选内容，再限制注入数量和长度。

系统会优先注入以下几类内容：

* 用户长期画像摘要
* 关系规则候选
* 相关事件候选

这种“候选式注入”可以显著降低 token 浪费，同时提升上下文相关性。

### 10.7 Prompt 组织方式

在 Prompt 设计上，项目并不是使用一段固定提示词，而是采用动态拼装方式，把多个信息层组合成最终输入。

### 1）角色设定层

这一层定义 AI 的人设、语气、关系定位和表达习惯，是系统保持角色一致性的基础。

### 2）记忆补充层

这一层提供和当前用户相关的长期信息与事件候选，帮助模型在多轮互动中体现“记得你”的效果。

### 3）执行结果层

当系统调用了联网搜索、提醒、动作等能力后，会把结构化结果作为补充信息注入模型，让模型基于现实结果组织自然语言回复。

### 10.8 调度流程

整个系统的核心链路可以概括为以下流程：

### 第一步：监听消息并生成事件

微信消息到达后，系统读取发送者、内容、会话信息，形成内部事件。

### 第二步：读取会话状态与上下文

运行时加载当前会话状态，读取最近对话历史，并判断当前是否存在未完成任务、待确认动作或待继续的工作流。

### 第三步：检索长期记忆

系统根据用户标识和当前语义场景，筛选可用于本轮对话的记忆候选，组装成记忆包。

### 第四步：意图判断与路由决策

决策层判断当前消息属于普通聊天、能力调用还是多步骤任务，并选择对应处理路径。

### 第五步：执行外部能力

若命中联网、提醒、微信动作等能力，系统先执行能力模块，并将结果保存为观察结果。

### 第六步：生成自然语言回复

模型网关将角色设定、短期上下文、长期记忆、能力结果整合后，生成最终回复。

### 第七步：写回上下文与状态

系统将本轮用户消息和 AI 回复写回短期上下文，同时更新运行时状态。

### 第八步：触发记忆沉淀与后台任务

当轮数达到阈值或触发特定条件时，系统进行长期记忆总结、提醒调度维护或其他后台更新。

### 10.9 配置与控制台

### 1）配置管理

可对模型参数、用户映射、交互策略、记忆相关设置、提醒策略、群聊策略等进行统一配置。

### 2）运行管理

支持启动、查看日志、检查状态、观察 API 调用趋势，降低调试和维护成本。

### 3）系统治理

支持模型档案管理、功能开关、动态设置写入等，使系统从“开发者脚本”转向“可持续运营工具”。

### 10.10 稳定性保障

### 1）异常捕获与降级

能力调用、驱动监听、模型请求、配置更新等关键环节都设置了异常捕获与返回规范，避免单点报错导致整个系统中断。

### 2）文件安全写入

记忆、上下文、动态配置等内容使用了临时文件写入、替换写回、文件锁等方式，减少并发写入或异常退出造成的数据损坏风险。

### 3）运行状态追踪

系统支持详细的流程日志记录，能够追踪路由、模型、记忆选择、工具执行等关键阶段，便于问题排查。

### 4）自动重启与心跳治理

项目具备定时重启、闲置重启、运行状态心跳等机制，适合桌面自动化这类容易受外部环境影响的运行场景。

### 5）敏感内容清理

系统对敏感内容和临时记忆清理设置了保护机制。
