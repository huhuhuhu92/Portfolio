## 6. 功能结构与产品方案

### 6.1 产品整体功能架构

本产品围绕“长期关系型情感陪伴”这一核心目标，构建以对话为中心、以记忆与关系演进为支撑、借助微信为输入输出平台的 Agent 系统。整体功能结构可分为四个核心模块：

1. **情感对话系统（核心交互层）**
2. **用户记忆与关系系统（长期价值层）**
3. **表达增强系统（情绪表达层）**
4. **Agent 能力扩展系统（工具能力层）**

四者共同构成一个具备“人格一致性 + 情感连续性 + 行为主动性”的陪伴型 Agent。

### 6.2 情感对话

### （1）自然语言对话能力

产品以即时对话为主要交互形式，通过大模型实现多轮上下文对话，支持：

* 日常闲聊（陪伴、吐槽、分享）
* 情绪疏导（安慰、共情、鼓励）
* 轻度角色互动（偏恋爱/暧昧语境）
* 轻任务指令理解（如提醒、生成内容等）

### （2）人格驱动对话

系统通过预设角色 Prompt（如固定人设、说话风格、情绪表达方式），使 Agent 具备：

* 稳定的人格特征（如温柔、占有欲、黏人等）
* 一致的语言风格（口语化、亲密称呼等）
* 情绪反应的“人设约束”

从而避免“工具感”，增强沉浸感。

### 6.3 用户记忆与关系

该模块是产品的核心差异点，用于实现“越聊越懂你”。

### （1）用户信息记忆

系统会持续记录用户相关信息，包括但不限于：

* 基本信息（昵称、偏好等）
* 生活事件（工作、学习、日常动态）
* 情绪状态（开心/难过/焦虑等）

并以结构化形式存储，供后续对话调用。

### （2）关系状态

通过独立的“关系记忆模块”，记录用户与 Agent 角色的互动关系，例如：

* 亲密度变化、关系进展
* 特殊事件（争吵、和好、纪念日）
* 情感标签（依赖、喜欢等）

使对话具备“关系连续性”，而非每次重置。

### （3）记忆写入与调用机制

* **写入触发** ：在关键对话节点（情绪表达、重要信息出现）自动记录
* **调用策略** ：在合适语境下主动引用历史信息（如“你上次不是说…”）

从而实现“被记住”的体验。

### 6.4 表达增强

为增强陪伴真实感，产品设计了多模态表达能力。

### （1）情绪表情系统

根据对话语义与情绪判断，动态匹配表情资源：

* 开心 / 生气 / 委屈 / 撒娇 / 疲惫 等情绪标签
* 对应 GIF / 图片自动发送

作用：

* 提升对话“拟人感”
* 增强情绪传达效率

### （2）情绪语气控制

通过 Prompt + 规则控制，实现：

* 语气变化（冷淡 / 撒娇 / 强势 / 温柔）
* 情绪波动（吃醋、生气、依赖等）

使 Agent 不再“永远温和”，而更接近真实人际互动。

### 6.5 Agent 工具能力

在情感陪伴之外，引入轻量工具能力，提升实用性与使用频率。

### （1）提醒与任务

支持用户通过对话触发提醒任务：

* 定时提醒（如“明天早上八点叫我起床”）
* 情绪型提醒（如“记得早点睡”）

通过后台任务机制执行，增强“被照顾感”。

### （2）图像生成

支持基于文本生成图片：

* 情绪表达（如“画一个我们在一起的场景”）
* 娱乐用途（头像、幻想画面等）

强化互动趣味性。

### （3）命令解析与调度

通过命令识别与意图判断：

* 区分闲聊 vs 指令
* 自动调用对应模块（提醒 / 生成 / 记忆等）

实现 Agent 的“行动能力”。

### 6.6 决策与调度

系统底层统一的决策流程：

1. **输入解析** （识别意图 + 情绪）
2. **上下文构建** （加载历史对话 + 记忆）
3. **能力判断** （是否调用工具）
4. **人格约束生成回复**
5. **后处理（表情 / 记忆写入 / 行为触发）**

## 7. 核心原型展示

### 7.1 原型章节说明

本章用于展示项目的核心原型设计，并说明每个关键页面在产品体系中的作用、承载的信息以及背后的设计逻辑。

这套原型围绕 **MemoMate 控制台** 展开，定位为一个面向 AI 关系陪伴型产品的后台配置与运营系统。它覆盖了从登录接入、角色初始化、用户与角色绑定、Prompt 管理、记忆管理、能力配置、互动策略、任务调度，到系统监控与异常恢复的完整链路。

本章采用“ **截图 + 页面说明 + 设计逻辑** ”的方式呈现。截图负责展示原型结果，文字负责解释产品逻辑。

### 7.2 原型整体定位

MemoMate 控制台系统的核心目标是帮助用户完成以下工作：

* 为不同用户配置专属一个/多个 AI 角色
* 管理 Prompt 与角色设定
* 维护用户画像与多层记忆
* 配置对话、搜索、视觉、生图等能力
* 设计互动节奏与主动触达策略
* 观察系统运行状态并处理异常情况

从产品视角看，这套原型解决问题：**当 AI 产品从单轮问答工具进入长期关系经营阶段后，后台系统应该如何组织能力、管理状态、支持运营。**

### 7.3 架构与模块

为了让系统结构清晰、配置路径明确，我将整套原型分为六个模块：

### 1. 接入

负责系统进入前的身份校验与初始配置。

* 登录页
* Onboarding 初始化流程

### 2. 控制台概览

负责提供系统总览和导航中枢。

* Dashboard 控制台首页

### 3. 角色与记忆

负责配置“这个 AI 在和谁互动、以什么身份互动、记住了什么”。

* 会话对象管理
* 用户画像
* 关系记忆
* 记忆总览
* 短期上下文
* 长期记忆

### 4. Prompt

负责管理角色设定与 Prompt 生产流程。

* Prompt 资产库
* Prompt 工坊
* AI 角色生成台

### 5. AI 能力配置

负责配置模型能力与外部能力接入。

* 对话引擎
* 辅助模型
* 视觉理解
* 图像生成
* 在线检索

### 6. 互动与运营

负责让系统可运营、可维护、可控制。

* 表情策略
* 群聊策略
* 节奏控制
* 主动触达
* 任务列表
* 调度总览
* 命令面板
* 实时日志
* 自恢复计划
* 控制台设置

这样的架构设计：

第一，便于后台管理者快速理解系统逻辑。

每个模块都有清晰边界，进入系统后可以迅速判断并定义问题在哪。

第二，便于后续产品扩展。

当系统增加新的能力时，可以直接挂载到现有架构中，不会破坏整体导航逻辑。

### 页面内容

该页面围绕“关系”展开，强调用户与 AI 的情感连接、信任变化、互动动态等内容。

### 设计逻辑

在很多 AI 产品里，记忆只停留在“用户喜欢什么”“用户做过什么”这类事实层信息。

这样做有助于把“内容记忆”和“关系记忆”分开管理，也更符合陪伴产品的长期互动特点。

对于目标用户来说，AI 的价值往往不只来自回答是否准确，也来自关系是否连贯。

关系记忆模块正是支撑这种连贯感的重要部分。

### 7.4.8 短期上下文

![Pasted image 20260331095707](../img/Pasted%20image%2020260331095707.png)

【短期上下文 截图】

### 页面目标

短期上下文页用于管理当前会话阶段内的即时语境信息，支撑 AI 对连续对话的理解。

### 页面内容

页面聚焦当前会话中的临时记忆，例如近期话题、手动触发上下文摘要等。

### 设计逻辑

短期上下文的特点是变化快、生命周期短、对当前回复影响强。

因此它应当独立于长期画像和关系记忆之外存在。

将它单独展示有几个好处：

* 让用户看清当前回复依赖了哪些临时信息
* 让系统的上下文来源更透明
* 便于在需要时清空或重置当前会话状态

### 思考

在实际 AI 产品中，很多异常回复都来自上下文管理失控。

所以短期上下文页面不仅是一个展示页，也是在表达产品对“上下文可控性”的重视。

它让会话记忆的来源更可解释。

### 7.4.9 长期记忆

![Pasted image 20260331095953](../img/Pasted%20image%2020260331095953.png)

【长期记忆 截图】

### 页面目标

长期记忆页用于沉淀用户跨会话稳定存在的信息，是角色长期陪伴能力的重要基础。

### 页面内容

这部分被命名为“{AI角色名}日记”，  页面中承载的是日记条数，日记具体内容，日记可手动编辑等功能。

### 设计逻辑

长期记忆与短期上下文的最大差异在于：它不追求即时变化，而追求长期稳定。

因此页面设计更强调“记录、归档、沉淀”的感觉。

使用“日记”这一表达，有两个目的：

1. 让记忆系统更拟人，更符合陪伴场景
2. 让用户更沉浸，能从另一个视角看见这段关系

### 思考

长期记忆本质上是结构化知识沉淀，但在陪伴产品里，它需要以更有人情味的方式呈现。

因此我用“角色日记”的表达承接技术能力，同时保留产品体验上的温度感。

### 7.4.10 Prompt 资产库

【 Prompt 资产库 截图】

### 页面目标

Prompt 资产库用于管理已经已有的的角色设定，支持编辑、查看与导出。

### 页面内容

页面中展示多个 Prompt 资产项，每项包含：

我把它设计成资产库，目的是让 Prompt 从一次性配置变成可持续运营的内容资源。

### 7.4.14 辅助模型配置

![Pasted image 20260331170143](../img/Pasted%20image%2020260331170143.png)

【辅助模型配置截图】

### 页面目标

辅助模型页用于配置用于后台任务的小模型，也是整个agent的大脑模型，如意图识别、情感分析、记忆提取、摘要生成等。

### 页面内容

页面辅助任务拆分为多个场景：

* 意图识别
* 情感分析
* 记忆提取
* 摘要生成

每个场景都可以配置相应的模型与参数。

### 设计逻辑

主对话模型承担的是用户可见的核心回复能力，而后台还存在大量支撑型任务。这些任务不一定需要由主模型完成。将其拆分给辅助模型，有助于：

* 降低成本
* 提高效率
* 增强模块独立性

因此，这个页面表达的是一种“任务分工”思路：不同模型负责不同职责，系统通过编排将它们协同起来。

### 产品思考

 在AI 系统架构层中，一个成熟的 AI 产品，通常不依赖单一模型完成所有工作，而是通过模型分层来平衡成本、延迟与效果。

辅助模型页正是遵循了这种架构思路。

### 7.4.15 在线检索

![Pasted image 20260331170405](../img/Pasted%20image%2020260331170405.png)

【在线检索 截图】

### 页面目标

在线检索页用于配置系统联网搜索能力，使 AI 在需要外部信息时具备检索与补充事实的能力。

### 页面内容

页面包括服务配置、调用开关、相关参数设置等，用于管理检索能力的接入和使用边界。

### 设计逻辑

长期陪伴型 AI 虽然依赖角色一致性和记忆体系，但也会遇到需要外部信息补充的场景。

因此，系统不能只依赖静态模型知识，还需要具备适度联网能力。

将搜索能力单独配置，而不是混在对话引擎页面中，有两个原因：

1. 搜索属于外部能力接入，有自己的调用成本和风险边界
2. 搜索能力需要单独控制是否开启、在什么场景下开启

### 思考

AI 陪伴产品既需要稳定角色感，也需要在必要时具备外部事实补充能力。

在线检索模块帮助系统在温度感和信息准确性之间取得平衡。

### 页面内容

页面中可配置服务商、API 节点、API Key、视觉模型等信息，用于定义图片解析链路。

### 设计逻辑

这种结构表达更符合多模态系统的设计逻辑。

### 思考

多模态不是简单增加一个功能按钮，而是需要在后台中拥有可独立配置、可单独管理的能力模块。

### 7.4.20 调度总览

![Pasted image 20260331171558](../img/Pasted%20image%2020260331171558.png)

【调度总览 截图】

### 页面内容

页面围绕任务调度、执行节点、状态总览等信息展开，用于帮助用户理解系统行为如何按计划发生。

### 7.4.21 表情策略 / 群聊策略 / 节奏控制

![Pasted image 20260331171659](../img/Pasted%20image%2020260331171659.png)

### 页面目标

这组三个页面共同服务于“互动风格控制”，用于让 AI 的行为更贴近真实互动语境。

### A. 表情策略 Emoji Strategy

### 页面作用

控制 AI 在什么语境下使用表情，以及表情的频率和风格。

### 设计逻辑

表情不是装饰性元素，而是语气的一部分。

在陪伴型场景中，表情的使用频率会直接影响角色是否显得自然、亲近或轻浮。

因此需要将其纳入策略控制，而不是交给模型自由发挥。

### B. 群聊策略 Group Chat Strategy

### 页面作用

控制 AI 在群聊环境中的触发逻辑，例如被 @ 时是否回复、回复概率、参与条件等。

### 设计逻辑

群聊与私聊的互动规则完全不同。

系统在群聊中需要更谨慎地判断何时发言、如何发言，否则会显得打扰或失控。

页面中通过规则化方式设置触发条件，体现的是“场景化策略管理”。

### C. 节奏控制 Rhythm Control

### 页面作用

控制 AI 回复的节奏感，包括思考延迟、消息节奏、停顿感等。

### 设计逻辑

真实感不只来自回复内容，也来自回复时机。

如果系统始终秒回、节奏一致，用户很容易感到机械。

因此通过节奏控制模块，系统可以模拟更自然的交流节拍，提升陪伴感和拟人感。

### 思考

AI 产品体验不仅是内容设计，也是行为设计。

语言、表情、节奏、群聊触发，这些因素一起决定了角色是否成立，互动是否可信。

### 7.5 页面之间的核心流程逻辑

为了让整套原型形成闭环，我将各页面串联为一条清晰的后台工作链路：

### 第一步：进入系统并完成初始化

用户从登录页进入控制台，在 Onboarding 中完成引擎连接、对象配置和基础策略设定，建立系统运行的最小闭环。

### 第二步：确定服务对象与角色关系

在会话对象管理中完成用户与角色绑定，在用户画像、关系记忆、记忆总览中逐步建立系统对用户的长期理解框架。

### 第三步：生产并管理角色设定

通过 Prompt 资产库查看已有角色资产，在 Prompt 工坊中编辑优化，在 AI 角色生成台中快速生成初稿，构成完整的 Prompt 生产链路。

### 第四步：接入 AI 能力并配置系统能力边界

通过对话引擎、辅助模型、在线检索、视觉理解、图像生成等页面，配置系统的基础能力与扩展能力。

### 第五步：设计互动方式与主动行为

通过表情策略、群聊策略、节奏控制、主动触达、任务列表和调度总览，进一步定义系统行为方式，让 AI 从回答问题走向关系陪伴。

### 第六步：进入运营与稳定性管理

通过命令面板进行快速控制，通过实时日志观测运行情况，通过自恢复计划处理异常，通过控制台设置管理系统基础安全。

### 7.6 章节结论

这套原型展示的是一个 AI 陪伴型产品从角色定义、记忆管理、能力调用到运营监控的完整后台逻辑。

在设计过程中，我关注的是各模块之间是否形成了稳定的产品闭环。通过页面层的具体设计，体现我对 AI 产品系统结构、角色管理、Prompt 工程、记忆机制、主动行为与可运营性的综合理解。
